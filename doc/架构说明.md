# 架构说明

## 模块划分

cccc主要划分为以下几个模块：

- GPU管理：GPU设备的选择和管理，GPU基础计算库（blas，cudnn，miopen等）的上下文创建等。NVidia和AMD的GPU均支持。
- 数据管理：数据在CPU和GPU上的分配，以及它们之间的传输和存储。
- 矩阵：矩阵类和基础计算，深度学习相关的扩展计算。
- 矩阵操作：记录操作类型，输入矩阵，输出矩阵，参数等，调用矩阵的计算来实现网络的正向和反向。
- 网络结构：定义网络结构（即有序的操作列表），损失函数等。也包含网络结构的解析和保存，单次训练和测试数据。
- 准备器：用于数据的加载预处理和变换，仅处理CPU部分。
- 工作流：将以上模块整合，并处理网络和准备器之间的数据传递。

cccc并不包含图像处理等功能，一般由准备器模块通过调用外部库实现。

如有需要，网络结构也留出了调用外部库的接口。

## 矩阵的内存管理

一般的矩阵赋值，为浅拷贝，即副本共享内存。且无论原矩阵与副本如何变化，二者始终共享内存。一般来说，该空间不会变小。

若强制将分配空间变小，可能引发未知访问错误，非特殊需求请勿这样做。

反向数据由矩阵自己管理。

计算所需的工作空间一般来说有GPU管理器持有，公用以节省显存。如需要不止一个矩阵，或需要保留中间结果，则由计算结果矩阵持有。

## 数据的训练过程

准备器会读取和处理整个epoch的数据，并将其存储在CPU内存中。

工作流会将这些数据按照GPU个数分块，上传到不同GPU上进行训练。每次训练一个batch的数据，再移动数据的指针到一个batch，直到整个epoch训练完成。

若不能整除，则最后的batch会包含剩余的数据，可能小于batch大小。

因此显存的占用为：一个epoch的数据（在不同GPU上分块存储） + 每个GPU上一个batch数量的网络结构，含前向和反向的所需要的工作空间。

一个epoch处理完之后，准备器会重新读取数据，工作流会继续进行下一个epoch的训练。

## 网络结构

网络结构包含一个矩阵操作的顺序列表，以及损失函数的列表。里面的矩阵均以共享指针的形式存储。

首个操作的首个输入矩阵视为X，最后一个操作的最后一个输出矩阵视为A。

若输入的矩阵与X无关，但与A有关，即认为是网络的权重矩阵。

## GpuControl

`GpuControl` 是 GPU 设备的管理类，每个物理设备对应一个实例，实例总数等于系统中检测到的 GPU 数量，在首次选中某设备时自动初始化，**不允许用户手动创建**。

该类持有与当前设备关联的底层计算库上下文：

| 后端 | 句柄 |
|------|------|
| NVIDIA CUDA | `Cublas*`（cuBLAS）、`cudnnContext*`（cuDNN） |
| AMD HIP/ROCm | `Rocblas*`（rocBLAS）、`miopenHandle*`（MIOpen） |

每个 `Matrix` 对象都持有一个 `GpuControl*` 指针，所有矩阵计算均通过该指针访问对应设备的计算库。

`GpuControl` 还负责以下职责：

- **显存分配与追踪**：`malloc` / `free` 包装底层 CUDA/HIP 分配接口，并将所有分配记录在静态的 `malloc_map_` 表中，便于调试和泄漏检测。
- **算法选择**：cuDNN 卷积算法的选择逻辑在此处集中管理（启发式 `Get*_v7` 接口，避免 `Find*` 接口的显存积累问题，详见下文）。
- **激活阶段**：`active_phase_` 字段区分训练模式（`ACTIVE_PHASE_TRAIN`）和推理模式（`ACTIVE_PHASE_TEST`），影响 BatchNorm、Dropout 等的行为。

多 GPU 训练时，工作流按设备数量将数据分块，每块数据上传到对应设备并在该设备的 `GpuControl` 上下文中完成计算。

## Matrix（矩阵类）

`Matrix` 是 cccc 所有数值计算的基础数据结构，同时支持二维矩阵和四阶张量两种形式：

- **二维矩阵**：行数 × 列数（`row_ × number_`），用于全连接、矩阵乘等。
- **四阶张量**：宽 × 高 × 通道 × 批（`W × H × C × N`），用于卷积、池化等图像操作。四阶张量的前三维展开后即为矩阵的一列（列优先存储，与 cuBLAS 一致）。

**浅拷贝语义**：矩阵赋值和拷贝构造均为浅拷贝，多个 `Matrix` 对象可以共享同一块底层内存（由 `std::shared_ptr<MatrixData>` 管理），修改任意一个的数据会影响所有共享副本。若需要独立副本，应显式调用 `clone()`。

**梯度矩阵**：每个 `Matrix` 持有一个延迟创建的同形梯度矩阵（`d_this_`），通过 `m.d()` 访问。梯度由矩阵自身负责分配，只有 `need_back_ = true` 时才会在反向计算中更新。对于冻结层的权重或网络输入，可将 `need_back_` 置为 false 以跳过梯度计算，节省显存和计算量。

**工作空间**：部分算子（如 BatchNorm、Attention）需要在正向和反向之间保留中间结果，这些结果存储在输出矩阵的 `workspace_` 向量中，随输出矩阵生命周期管理，不在 `GpuControl` 全局工作空间中。

**标记字段**：

| 字段 | 含义 |
|------|------|
| `is_weight_` | 是否为可训练权重（影响保存/加载和求解器更新） |
| `is_input_` | 是否为网络输入（影响计算图剪枝） |
| `need_load_` | 载入权重文件时是否加载该矩阵（用于部分冻结预训练权重） |
| `weight_name_` | 权重的具名标识，用于按名称加载预训练参数 |

## MatrixOp（矩阵操作）

`MatrixOp` 是网络中单个算子的描述单元，记录一次前向/反向可复现计算所需的全部信息：

- **操作类型**（`MatrixOpType`）：枚举全部支持的算子，包括 `ADD`、`MUL`、`CONV`、`POOL`、`ACTIVE`、`LAYER_NORM`、`RMS_NORM`、`ATTENTION`、`EMBED`、`KV_CACHE`、`ROPE` 等共约 30 种。
- **输入/输出矩阵列表**（`in_` / `out_`）：均以 `shared_ptr<Matrix>` 存储，网络中的矩阵共享关系完整保留。
- **标量参数**（`a_` / `b_`）：缩放系数等浮点参数；`window_` / `stride_` / `padding_` 用于卷积和池化。
- **扩展参数**（`anys_`）：类型不固定的额外参数（如激活函数类型、GroupNorm 分组数等）。

`MatrixOp` 的静态方法 `forward(op_queue)` 和 `backward(op_queue, loss, clear_d)` 分别顺序/逆序遍历操作列表，调用每个算子的 `forwardData()` / `backwardDataWeight()`，构成完整的训练迭代：

```
forward:   op[0] → op[1] → … → op[N-1]
backward:  loss[k] → … → op[N-1] → … → op[0]
```

**计算图剪枝**：`checkConnect()` 分析各算子与网络输入 X、输出 A 以及损失函数的连通性，在推理时剪去与损失无关的反向路径（`connect_loss_ = false`），在冻结层训练时剪去与 X 无关的前向路径（`connect_x_ = false`），减少不必要的计算。

`Net` 类持有一个 `std::vector<MatrixOp> op_queue_`，这个列表由 cifa 脚本解析时逐算子追加构建，一旦构建完成即不再修改，可重复用于多次前向/反向迭代。

## 待改进：epoch 级数据传输与计算并行（双缓冲）

### 现状与瓶颈

当前训练循环中，`trainOneNet` 每个 epoch 开始时通过 `Matrix::copyRows`（内部调用同步 `cudaMemcpy`）将 CPU 数据一次性拷贝到 GPU，拷贝期间 GPU 完全空闲。
CPU 内存使用普通 `malloc`（可分页内存），DMA 传输前需先做一次内部 pageable→pinned 中间拷贝，效率约是 pinned memory 的一半。

曾尝试将 `copyRows` 移到数据准备线程与 GPU 计算并行，但所有 CUDA 操作均在 legacy default stream（stream 0）上执行，default stream 的隐式同步规则导致两线程的操作仍严格串行，反而引入额外跨线程开销，比原来更慢。

### 改进方案

**代价**：`X_train_gpu` / `Y_train_gpu` 各申请两份（显存占用翻倍），其余代码不变。

**步骤**：

1. **`MatrixData::copyByByteAsync`**（已添加）：`cudaMemcpyAsync` 重载，stream 以 `void*` 传入。

2. **在 `trainOneNet` 中添加 copy stream 和双缓冲**（暂未实现）：

```cpp
// 初始化阶段（新增）
cudaStream_t copy_stream;
cudaStreamCreateWithFlags(&copy_stream, cudaStreamNonBlocking);  // 非阻塞！不与 default stream 互相等待

int cur = 0;
Matrix X_bufs[2] = {x_dim, x_dim};
Matrix Y_bufs[2] = {y_dim, y_dim};

// 首 epoch 同步 copy（无上一 epoch 可并行）
WAIT_UNTIL(ti.data_prepared == epoch_count || ti.stop);
epoch_count++;
Matrix::copyRows(data_preparer_->X, 0, X_bufs[cur], 0, X_bufs[cur].getNumber());
Matrix::copyRows(data_preparer_->Y, 0, Y_bufs[cur], 0, Y_bufs[cur].getNumber());
++ti.data_distributed;

// 主循环
while (ti.stop == 0) {
    int next = 1 - cur;
    bool async_issued = false;

    for (int iter = 0; iter < num_iters; iter++) {
        // 训练当前 epoch（不变）
        net->getX().shareData(X_bufs[cur], 0, iter * net->getX().getNumber());
        net->getY().shareData(Y_bufs[cur], 0, iter * net->getY().getNumber());
        net->active(...);

        // 新增：检测到下一 epoch 数据就绪，立刻发起 async copy
        if (!async_issued && ti.data_prepared >= epoch_count) {
            // 需要 Matrix::copyRowsAsync 封装（调用 copyByByteAsync）
            copyRowsAsync(data_preparer_->X, X_bufs[next], copy_stream);
            copyRowsAsync(data_preparer_->Y, Y_bufs[next], copy_stream);
            async_issued = true;
        }
    }
    // 若训练结束前数据还未就绪，等待后再发起 async copy
    if (!async_issued) {
        WAIT_UNTIL(ti.data_prepared >= epoch_count || ti.stop);
        copyRowsAsync(data_preparer_->X, X_bufs[next], copy_stream);
        copyRowsAsync(data_preparer_->Y, Y_bufs[next], copy_stream);
    }
    cudaStreamSynchronize(copy_stream);  // 确保 CPU buffer 此后可安全覆盖

    epoch_count++;
    ++ti.data_distributed;
    cur = next;
}

// 析构时销毁 stream
cudaStreamDestroy(copy_stream);
```

3. **可选：pinned memory**：将 `data_preparer_->X/Y` 的分配改用 `cudaMallocHost`，DMA 无需中间缓冲，async copy 速度约提升 2 倍。

### 预期收益

- `prepareData` 耗时 ≥ copy 耗时时：copy 完全隐藏，epoch 总时间不变
- `prepareData` 较快时：节省约 copy 时间的一半至全部
- 典型分类训练（5000 样本 × 128²×3，PCIe 3.0）：copy ≈ 50 ms / epoch，训练 ≈ 5~30 s / epoch，绝对收益小；但若 prepareData 含磁盘 I/O，收益可能显著

## 权重合并（combineWeights）

`Net::initWeights()` 完成后，如果当前为训练模式（`train_epochs > 0`），会立即调用 `combineWeights(weights_, all_weights_)`。

**合并的目的**：将所有可训练权重矩阵（`weights_` 列表中 `needBack() == true` 的成员）的数据按 256 字节对齐拼接到一块连续 GPU 内存 `all_weights_`，同时将各权重矩阵的数据指针重定向到这块内存的对应位置。Solver 在每个 batch 训练结束时只需对 `all_weights_` 调用一次 `updateWeights()`，即可批量更新全部权重，而无需对每个矩阵单独调用——在多 GPU 同步和 Solver 内部向量运算中效率更高。

**不执行合并的情形**：

1. **推理模式**（`train_epochs = 0`）：Solver 不初始化，`combineWeights` 也不调用。权重由各矩阵各自持有，`all_weights_` 保持为空，节省显存。
2. **`train::separate_update_weight = 1`**：每个权重矩阵拥有独立的 `Solver` 实例（`solvers_[m]`），训练时逐矩阵单独调用 `solvers_[m]->updateWeights(*m, batch)`。此时 `all_weights_` 同样不使用（`solver_.init` 也不被调用）。该选项适用于需要对不同权重使用不同求解器类型或学习率的场景。

## cuDNN workspace 管理与算法选择

### FindAlgorithm 的显存陷阱

cuDNN 提供两类卷积算法选择接口：

- **`cudnnFindConvolution*Algorithm`**：穷举候选算法，对每个算法分配 workspace 实际运行计时，然后选最快的。这些 workspace **不会在调用完成后释放**，而是缓存在 cuDNN handle 中，以便下次同样尺寸/参数的卷积直接复用。
- **`cudnnGetConvolution*Algorithm_v7`**：通过启发式规则推断最佳算法，**不分配任何 workspace**，不做实际计算。

**问题**：使用 `Find*` 接口时，每层卷积（前向、反向 data、反向 filter 各一次）都会在 handle 缓存中积累 workspace。层数越多，累计越大，且在 handle 生命周期内不释放，严重压缩用于数据和激活的显存空间。

典型量级参考：
- **3 层 CNN 自编码器**（128×128 输入）：累计 workspace **~2.6 GB**
- **Transformer 自编码器**（128×128，2 个 Encoder + 2 个 Decoder，含多头注意力层卷积）：累计 workspace + 网络参数共 **~21 GB**。实测在 32 GB GPU 上，X_train=3.75 GB、Y_train=3.75 GB 已占 7.5 GB，剩余 ~24 GB 中 ~21 GB 被 workspace 占用，导致 OOM。

### cccc 的解决方案

cccc 已将所有卷积算法选择切换为 `Get*_v7` 接口（启发式，无 workspace 分配）。代价是所选算法可能不如 `Find*` 极致最优，但对大多数架构（尤其是批量较小的工业检测场景），性能差距在 5% 以内，而显存节省是决定性的。

若需要在特定平台上极致性能，可临时将 `GpuControl.cpp` 中的算法选择函数换回 `Find*`，但需确保 GPU 显存足够（建议 32 GB+）。
